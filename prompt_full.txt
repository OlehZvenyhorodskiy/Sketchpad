# Від баг-фіксу до академічного демонстраційного проєкту: Глибокий технічний аудит та розробка 15 "чіт-кодів" для Android-додатку Sketchpad

## Аналіз та виправлення критичних багів: Трансформації, продуктивність та жести

Ця секція представляє детальний технічний аудит та рішення п'яти ключових багів, що ідентифіковані в проєкті Sketchpad. Аналіз базується на розумінні внутрішньої роботи Jetpack Compose, API роботи з канвасом та системи обробки жестів. Кожен пункт містить опис проблеми, технічні причини її виникнення, а також готовий код для її виправлення.

**Баг 1: Блокування трансформації доданих зображень та фігур (Rotation / Scaling / Translation)**

Симптом полягає в тому, що об'єкти, додані на полотно, не реагують на жести повороту, масштабування або переміщення. Ця проблема є результатом конфлікту в обробці жестів між окремими компонентами UI та глобальним слоєм трансформації полотна. Додані фігури, ймовірно, мають власний `pointerInput` модифікатор для взаємодії, який конфліктує з `detectTransformGestures` або `rememberTransformableState`, що застосовується до самого `Canvas`. Коли користувач намагається взаємодіяти з фігурою, подія дотику може бути "захоплена" одним з обробників, але не передана іншому, що призводить до того, що жест не виконується очікуваною дією.

Технічно, проблема виникає через невірне розподілення відповідальності. Якщо система трансформації полотна активна, вона може сприймати всі жести масштабування та повороту, не передаючи їх фігурі, яка прагне до локальної трансформації. Для вирішення цього необхідно ввести поняття "активного об'єкта", який наразі перебуває в режимі редагування. Коли користувач виділяє фігуру, вона повинна "перевзяти" обробку жестів, деактивуючи глобальні трансформації для неї. Це досягається шляхом умовної вкладеності обробників жестів.

Нижче наведено виправлений псевдокод, що демонструє принцип. Припустимо, є стан `selectedShapeId`, який вказує на ID фігури, що редагується.

```kotlin
// Внутрішність InteractiveCanvas.kt
val transformationState = rememberTransformableState(onTransformChanged = { zoomChange, rotationChange, offsetChange ->
    // Оновлення глобальної матриці полотна
    viewModel.updateCanvasTransform(zoomChange, rotationChange, offsetChange)
}))

Canvas(modifier = Modifier
    .fillMaxSize()
    .then(
        if (viewModel.selectedShapeId != null) {
            Modifier.pointerInput(Unit) { detectTransformGestures(...) }
                // ... логіка для трансформації вибраної фігури
        } else {
            Modifier.transformable(state = transformationState)
        }
    )
) {
    // Малювання полотна...
}
```

**Баг 2: Просідання FPS до 2-5 при масштабуванні порожнього графіка**

Ця проблема є класичним прикладом циклу нескінченного переобрахування, що блокує головний потік. Проблема виникає, коли параметри для малювання сітки, такі як `gridStepX` та `gridStepY`, обчислюються безпосередньо в Composable-компоненті на основі поточної матриці трансформації, яка, в свою чергу, змінюється відповідно до жесту масштабування [[159]]. Це створює небезпечний зворотний зв'язок:
1.  Користувач починає жест масштабування.
2.  `onTransformChanged` оновлює матрицю трансформації.
3.  Composable-компонент, що малює сітку, виявляє, що його залежність (матриця) змінилася, і запускає переобрахування.
4.  Під час переобрахування перераховується `gridStep` на основі нової матриці.
5.  Це змінює вивідний результат, що призводить до нового виклику `onDrawContent`.
6.  Процес повторюється до моменту, коли жест завершується, витрачаючи всю CPU-потужність.

Ключове рішення — відокремити дорогі обчислення від UI-рендерингу за допомогою `drawWithCache` та правильно використовувати похідний стан. Логіка малювання сітки повинна бути обчислена один раз і закешована, а не виконуватися на кожному кадрі.

Виправлений код:

```kotlin
// Внутрішність InteractiveCanvas.kt
@Composable
fun GridCanvas(canvasSize: Size, transformState: TransformState, viewModel: SketchpadViewModel) {
    val paint = remember {
        SolidColor(Color.LightGray.copy(alpha = 0.3f))
    }.value

    // Обчислюємо крок сітки лише тоді, коли змінюється матриця трансформації
    val gridStep by remember(transformState.transformedMatrix) {
        derivedStateOf {
            // Приклад формули: крок сітки залежить від поточного масштабу
            val scale = sqrt(transformState.transformedMatrix determinant)
            (10f * scale).coerceAtLeast(1f)
        }
    }

    Canvas(modifier = Modifier.fillMaxSize()) {
        drawWithCache {
            // Кешуємо весь результат малювання
            onDrawBehind {
                val centerX = size.width / 2f
                val centerY = size.height / 2f
                
                // Перемістити початок координат у центр
                save()
                translate(centerX, centerY)
                
                // Застосувати трансформацію полотна
                concatTransform(transformState.transformedMatrix)
                
                // Перемалювати початок у (0,0) після трансформації
                translate(-centerX, -centerY)
                
                // Малювання сітки
                drawHorizontalLines(paint, gridStep, size)
                drawVerticalLines(paint, gridStep, size)
                
                restore()
            }
            
            // Вказуємо Compose, що кеш має бути перевідновлений, коли змінюється gridStep
            // або сама матриця трансформації
            onDispose {
                // ...
            }
        }
    }
}

// Функції для малювання ліній
private fun DrawScope.drawHorizontalLines(paint: Paint, step: Float, size: Size) {
    for (y in -step..size.height step + step) {
        drawLine(paint, Offset(0f, y), Offset(size.width, y))
    }
}

private fun DrawScope.drawVerticalLines(paint: Paint, step: Float, size: Size) {
    for (x in -step..size.width step + step) {
        drawLine(paint, Offset(x, 0f), Offset(x, size.height))
    }
}
```
Цей підхід значно покращує продуктивність, оскільки вся логіка малювання сітки кешується як один примітив, і перемалювання відбувається лише тоді, коли стан, що впливає на неї (масштаб), дійсно змінюється [[156,194]].

**Баг 3: Статичні аудіохвилі під час запису з мікрофона**

Проблема статичного аудіовізуалізатора найімовірніше пов'язана з механізмом отримання та обробки аудіоданих. Якщо використовується стандартний `Visualizer` API, він може не підтримувати запис з мікрофона на всіх пристроях, або дані надходять з недостатньою частотою для плавної анімації [[71]]. Якщо ж використовується `AudioRecord`, проблема може полягати в тому, що дані читаються в фоновому потоці, але оновлення стану для UI не синхронізується правильним чином, або корутина опитування не забезпечує достатню частоту (30-60 FPS).

Оптимальним рішенням для низьколатентного запису є використання бібліотеки **Oboe** від Google. Oboe — це C++ бібліотека, яка надає доступ до низькорівневого аудіо API Android (AAudio на нових версіях), забезпечуючи максимально низьку затримку та високу продуктивність, працюючи на 99% пристроїв з API 16+ [[253,254,255]]. Її можна легко інтегрувати в Kotlin-проєкт через Android NDK.

План реалізації:
1.  **Інтеграція Oboe:** Додати Oboe до проекту через Gradle i ndk.
2.  **Створення аудіо-лінії в ViewModel:** У `SketchpadViewModel` створити `MutableStateFlow<ByteArray>`, яка буде випускати аудіодані.
3.  **Реалізація запису:** У фоновому `CoroutineScope` створити клас-обгортку навколо Oboe, який налаштовує запис з мікрофона та у власному зворотному виклику читає дані з буфера.
4.  **Передача даних на UI:** У зворотному виклику Oboe оброблені дані (наприклад, макс. амплітуда) передаються на UI-потік за допомогою `stateFlow.tryEmit(...)`.

Виправлений код для `AudioWaveformVisualizer`:

```kotlin
// Внутрішність AudioWaveformVisualizer.kt
@Composable
fun AudioWaveformVisualizer(viewModel: SketchpadViewModel) {
    var waveformData by remember { mutableStateOf<List<Float>>(emptyList()) }
    
    LaunchedEffect(viewModel.audioDataFlow) {
        // Слухаємо потік даних з ViewModel
        viewModel.audioDataFlow.collect { audioBytes ->
            // Проста обробка: розрахунок амплітуди для кожного блоку байтів
            val amplitudes = audioBytes.chunked(4).map { chunk ->
                // Приклад розрахунку: взяти абсолютне значення першого int
                chunk.toInt(0).absoluteValue.toFloat()
            }
            waveformData = amplitudes
        }
    }
    
    Canvas(modifier = Modifier.fillMaxWidth().height(100.dp)) {
        if (waveformData.isNotEmpty()) {
            // Малюємо хвилю на основі отриманих даних
            drawWaveform(waveformData, size)
        }
    }
}

// Функція для малювання хвилі
private fun DrawScope.drawWaveform(data: List<Float>, canvasSize: Size) {
    val path = Path()
    val amplitudeScale = (canvasSize.height * 0.4f).coerceAtMost(100f)
    val xStep = canvasSize.width / data.size.coerceAtLeast(1).toFloat()
    
    path.moveTo(0f, canvasSize.height / 2f)
    for (i in data.indices) {
        val x = i * xStep
        val y = (data[i] / 32768f * amplitudeScale) + (canvasSize.height / 2f)
        path.lineTo(x, y)
    }
    path.lineTo(canvasSize.width, canvasSize.height)
    path.lineTo(0f, canvasSize.height)
    path.close()
    
    drawPath(path, color = Color.Blue.copy(alpha = 0.5f))
    drawPath(path, color = Color.Blue, style = Stroke(width = 2f))
}
```
Цей підхід гарантує, що аудіодані надходять до UI з максимально можливою швидкістю і високою точністю, що робить візуалізатор динамічним і реактивним.

**Баг 4: Зрив та "залипання" переміщення лінійки (Ruler Drag Bug)**

Ця проблема типова для обробки жестів переміщення та виникає через неправильне відстеження координат. Ймовірна причина полягає в тому, що при обробці події `ACTION_MOVE` розраховується зміщення відносно початкової точки дотику, але поточні координати не оновлюються. Це призводить до ситуації, коли кожен наступний `delta` розраховується від однієї й тієї ж початкової точки, що робить переміщення різким та "зривається". Також може бути проблема з втратою `PointerId` при переході від однієї події до іншої.

Найпростіше та найефективніше рішення — використати вбудований у Compose API `detectDragGestures`. Цей API абстрагується від всіх низькорівневих деталей, таких як управління `PointerId` та розрахунок `delta`, надаючи розробнику простий інтерфейс для обробки жесту перетягування.

Виправлений код для переміщення лінійки:

```kotlin
// Внутрішність Ruler.kt
@Composable
fun Ruler(rulerState: RulerState) {
    Box(
        modifier = Modifier
            .fillMaxSize()
            .pointerInput(rulerState.x, rulerState.y) {
                detectDragGestures(
                    onDragStart = { offset ->
                        // Жест почався
                    },
                    onDragEnd = {
                        // Жест завершився
                    },
                    onDragCancel = {
                        // Жест скасовано
                    },
                    onDrag = { change, dragAmount ->
                        // dragAmount — це вже готове зміщення для цього кроку
                        // Оновлюємо стан лінійки
                        rulerState.updatePosition(rulerState.x + dragAmount.x, rulerState.y + dragAmount.y)
                        
                        // Необхідно споживати зміну, щоб інші обробники не отримали її
                        change.consume()
                    }
                )
            }
    ) {
        // Малювання лінійки
        Text("📏 Ruler", Modifier.offset { IntOffset(rulerState.x.toInt(), rulerState.y.toInt()) })
    }
}

// State class для лінійки
data class RulerState(var x: Float, var y: Float) {
    fun updatePosition(newX: Float, newY: Float) {
        x = newX
        y = newY
    }
}
```
Використання `detectDragGestures` робить код чистішим, менш піддавальним помилкам та забезпечує плавне та стабільне переміщення [[175,242]].

**Баг 5: Неконтрольоване збільшення лінійки при спробі повороту (Scale Distortion)**

Ця проблема виникає через неправильне розділення жестів масштабування та повороту в `detectTransformGestures`. Цей API повертає об'єкт `TransformEvent`, який містить інформацію про зміну масштабу (`zoom`), зміну кута (`rotation`) та зміщення (`pan`). Коли користувач виконує двофігурний жест повороту, система одночасно отримує ненульове значення `rotation` та, часто, ненульове значення `zoom` (навіть якщо воно мало). Якщо логіка обертання та масштабування не є взаємно виключними, то спроба застосувати обидва перетворення до лінійки призводить до її неконтрольованого збільшення.

Щоб вирішити цю проблему, необхідно розділити векторну математику. Кут обертання слід обчислювати окремо від довжини вектора. Кут можна знайти за допомогою `atan2`, а довжину — за допомогою `hypot`. Потім треба застосовувати лише обертання, якщо жест є чистим поворотом.

Виправлений код для обробки жесту лінійки:

```kotlin
// Внутрішність Ruler.kt
@Composable
fun RotatableRuler(rulerState: RulerState) {
    val touchSlop = ViewConfiguration.get(LocalContext.current).touchSlop
    
    Box(
        modifier = Modifier
            .fillMaxSize()
            .pointerInput(rulerState.rotation, rulerState.length) {
                detectTransformGestures(
                    onGestureStarted = { /* ... */ },
                    onTransformed = { fullZoomChange, fullRotationChange, panChange ->
                        // Розділяємо жести
                        
                        // 1. Жест масштабування (pinch): змінює довжину лінійки
                        // Перевіряємо, чи зміна масштабу значна
                        if (fullZoomChange != 1f && abs(fullZoomChange - 1f) > 0.01f) {
                            val newLength = rulerState.length * fullZoomChange
                            rulerState.updateLength(newLength.coerceAtLeast(10f)) // Мінімальна довжина
                        }
                        
                        // 2. Жест повороту (two-finger rotate): змінює кут
                        // Перевіряємо, чи зміна масштабу близька до 1 (тобто це не масштабування)
                        if (abs(fullZoomChange - 1f) < 0.01f && abs(fullRotationChange) > 0.01f) {
                            rulerState.updateRotation(rulerState.rotation + fullRotationChange)
                        }
                        
                        // 3. Жест перетягування (one-finger pan): переміщує лінійку
                        // Це просто оновлення позиції, яке не залежить від трансформацій
                        // rulerState.updatePosition(...)
                    },
                    gesturePriority = GesturePriority.Low // Надаємо перевагу іншим жестам, якщо потрібно
                )
            }
    ) {
        // Малювання трансформованої лінійки
        Box(
            modifier = Modifier
                .offset { IntOffset(rulerState.x.toInt(), rulerState.y.toInt()) }
                .graphicsLayer(
                    rotationZ = rulerState.rotation,
                    // scale х і y не використовуються для зміни довжини, бо вона задана в state
                )
                .size(rulerState.length.dp, 10.dp) // Висота лінійки
                .background(Color.Red)
        )
    }
}
```
У цьому рішенні ми явно розрізняємо три типи жестів: масштабування (змінює довжину), поворот (змінює кут) та перетягування (змінює положення). Це усуває конфлікт та забезпечує очікувану поведінку для кожної дії [[30,32]].

## Архітектурний синтез: Управління станом та оптимізація продуктивності

Аудит проєкту Sketchpad виявив, що більшість технічних проблем, зокрема трансформації та продуктивність, мають глибокі корені в архітектурі управління станом. Неналежне розподілення стану між `ViewModel` та локальним станом Jetpack Compose є ключовим архітектурним ризиком, який призводить до неефективних переобрахунків, неконсистентності даних та поганої UX. Цей розділ присвячений аналізу цього ризику та представленню стратегій для його мінімізації.

Основна проблема полягає в тому, що стан, який повинен пережити перезапуск активності (наприклад, контент полотна, історія дій), повинен зберігатися в `ViewModel` [[185,218]]. Однак, UI-спеціфічний стан, який не має сенсу зберігати (наприклад, поточний масштаб, положення камери, стан виділення фігури), краще тримати локально в Composable-компонентах за допомогою `remember` [[76]]. Коли цей розподіл порушується, наприклад, через занадто широкий `State` в `ViewModel`, що містить всі параметри відображення, це призводить до масового переобрахування UI, коли змінюється одна, навіть незначна, частина стану.

**Ризик №1: Масові переобрахування через неправильне управління станом**

Коли стан трансформації полотна (матриця) зберігається в `ViewModel` як `MutableStateFlow`, будь-яка зміна в ній (наприклад, при масштабуванні) призведе до переобрахування *всіх* Composable-компонентів, які спостерігають за цим станом. Якщо в композіції є елементи, які не залежать від матриці трансформації (наприклад, палітра кольорів, інструменти), вони також будуть переобраховуватися зайвий раз, створюючи вузькі місця [[157,266]].

**Стратегія оптимізації №1: Розділення стану та використання `derivedStateOf`**

Необхідно чітко розділити стан на дві категорії:
1.  **Бізнес-стан:** Дані, що мають значення для додатку (полотно, штрихи, історія дій). Це повинен бути `State` в `ViewModel`.
2.  **UI-стан:** Стан, що описує *як* ці дані відображаються (масштаб, орієнтація, виділені елементи). Цей стан повинен бути локальним, використовуючи `remember`.

Для стану, який є результатом дорогої обчислювальної операції на основі іншого стану, використовується `derivedStateOf`. Це дозволяє Compose кешувати результат і переобраховувати його лише тоді, коли його залежності справді змінюються [[78]].

Приклад реалізації:
```kotlin
// ViewModel
class SketchpadViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(SketchpadUiState())
    val uiState: State<SketchpadUiState> = _uiState
    
    data class SketchpadUiState(
        val canvasTransform: Matrix = Matrix(),
        val strokes: List<Stroke> = emptyList(),
        val selectedTool: Tool = PenTool()
        // ... інші поля бизнес-стану
    )
    
    // Функція для оновлення трансформації
    fun updateCanvasTransform(delta: Matrix) {
        _uiState.value = _uiState.value.copy(
            canvasTransform = _uiState.value.canvasTransform.concat(delta)
        )
    }
}

// Composable Screen
@Composable
fun SketchpadScreen(viewModel: SketchpadViewModel = hiltViewModel()) {
    // Спостерігаємо за загальним станом UI
    val uiState by viewModel.uiState.collectAsState()
    
    // Локальний UI-стан, що не впливає на інший UI
    var localUiState by remember { mutableStateOf(LocalUiState()) }
    
    // Похідний стан: матриця для малювання
    val drawMatrix by remember(uiState.canvasTransform) {
        derivedStateOf {
            // Деякі обчислення на основі canvasTransform
            uiState.canvasTransform.clone()
        }
    }
    
    // Похідний стан: список видимих штрихів
    val visibleStrokes by remember(uiState.canvasTransform, uiState.strokes) {
        derivedStateOf {
            // Фільтрація штрихів на основі видимої області
            uiState.strokes.filter { it.isVisible(uiState.canvasTransform) }
        }
    }
    
    // ... використання drawMatrix та visibleStrokes ...
}
```
Цей підхід дозволяє Compose точно визначити, які саме частини UI потребують оновлення, значно зменшуючи кількість переобрахунків [[129]].

**Ризик №2: Неконсистентність стану при асинхронних операціях**

Коли стан оновлюється як з боку `ViewModel` (через користувацьку дію), так і з боку UI (через жест), можливий стан гонитви. Наприклад, користувач починає масштабувати полотно, але `ViewModel` одночасно оновлює свій стан через іншу подію. Це може призвести до того, що UI відобразить стан, який уже не є актуальним, або до "зависання" жесту.

**Стратегія оптимізації №2: Використання Unidirectional Data Flow (UDF) та `SideEffects`**

Шаблон UDF передбачає, що єдиним джерелом істини для стану додатку є `ViewModel`. Події від користувача (жести, натискання кнопок) надсилаються в `ViewModel` у вигляді `Events`, які той обробляє та оновлює `State`. UI тільки відтворює стан, наданий `ViewModel`.

Для асинхронних операцій, які не повинні мутувати основний стан, але виконують побічні дії (наприклад, запис аудіо, збереження файлу), використовуються `SideEffects` (`LaunchedEffect`, `DisposableEffect`). Це гарантує, що побічні дії не викличуть зайвого переобрахування основного стану.

Приклад реалізації UDF:
```kotlin
// ViewModel
sealed class SketchpadEvent {
    object OnCanvasScaled : SketchpadEvent()
    data class OnStrokeAdded(val stroke: Stroke) : SketchpadEvent()
    // ... інші події
}

class SketchpadViewModel(...) : ViewModel() {
    private val _uiState = MutableStateFlow(SketchpadUiState())
    val uiState: State<SketchpadUiState> = _uiState
    
    private val _uiEventChannel = Channel<SketchpadEvent>()
    val uiEventChannel = _uiEventChannel.receiveChannel()

    fun handleEvent(event: SketchpadEvent) {
        viewModelScope.launch {
            _uiEventChannel.send(event)
        }
    }
    
    init {
        // Корутина, що слухає події та оновлює стан
        viewModelScope.launch {
            _uiEventChannel.receiveChannel().consumeEach { event ->
                when (event) {
                    is SketchpadEvent.OnCanvasScaled -> {
                        _uiState.value = _uiState.value.copy(
                            canvasTransform = _uiState.value.canvasTransform.scale(...)
                        )
                    }
                    is SketchpadEvent.OnStrokeAdded -> {
                        _uiState.value = _uiState.value.copy(
                            strokes = _uiState.value.strokes + event.stroke
                        )
                    }
                }
            }
        }
    }
}

// Composable Screen
@Composable
fun SketchpadScreen(viewModel: SketchpadViewModel = hiltViewModel()) {
    val uiState by viewModel.uiState.collectAsState()
    val context = LocalContext.current
    
    // Логіка запису аудіо виконується як побічна дія
    LaunchedEffect(Unit) {
        startAudioRecording(context) { amplitudeData ->
            // Оновлення стану аудіо
            viewModel.handleEvent(SketchpadEvent.AudioAmplitudeUpdate(amplitudeData))
        }
    }
    
    // Жести також генерують події
    Box(
        modifier = Modifier
            .pointerInput(Unit) {
                detectTransformGestures { _, _, pan, _ ->
                    viewModel.handleEvent(SketchpadEvent.CanvasPanned(pan))
                }
            }
    ) {
        // ...
    }
}
```
Цей підхід робить потік даних предсказуемим і контролюваним, усуваючи більшість проблем, пов'язаних з неконсистентністю стану [[258]].

**Ризик №3: Великі об'єкти стану та проблеми зі збереженням**

Якщо `ViewModel` містить величезні об'єкти стану (наприклад, великі масиви штрихів або зображення), це може призвести до проблем з використанням оперативної пам'яті та проблем зі збереженням/відновленням стану під час перезапуску процесу.

**Стратегія оптимізації №3: Використання `rememberSaveable` та Lazy Loading**

Для стану, який потрібно зберегти, використовується `rememberSaveable`. Важливо передавати туди лише ті дані, що необхідні для відтворення UI після перезапуску. Для великих об'єктів, які не потрібно зберігати (наприклад, тимчасові обчислювальні результати), вони повинні бути видалені з `ViewModel` після використання.

Для великих наборів даних (наприклад, історія дій) можна реалізувати ліниве завантаження. `ViewModel` може зберігати лише ID або посилання на дані, а самі дані завантажуються та кешуються лише тоді, коли вони потрібні для відображення [[220]].

Використання `rememberSaveable` для критичних UI-компонентів, як-от `rulerState` або `cameraTransform`, є відповідальністю Composable-компонента, а не `ViewModel`. `ViewModel` повинен зберігати лише бізнес-логіку та дані [[127]].

Застосування цих архітектурних принципів — розділення стану, UDF, та розумне кешування — є ключовим для перетворення Sketchpad на стабільний, продуктивний та масштабований додаток, готовий до впровадження складних "чіт-кодів".

## Каталог "Чіт-кодів" №1: Інтелектуальна геометрія та математичні рушії

Цей розділ присвячений розробці трьох просунутих функцій, які виводять додаток Sketchpad на новий рівень, перетворюючи його на інструмент для студентів комп'ютерних наук, спеціалісти на математиці та геометрії. Кожна функція включає глибокий технічний опис, вибір технологічного стеку та алгоритмічну модель.

**Фіча 1: Smart Shape Recognizer & Vectorizer**

*   **Назва та суть:** Автоматичне розпізнавання накреслених від руки геометричних фігур (коло, прямокутник, трикутник) та їх перетворення на точні векторні об'єкти з можливістю редагування.
*   **Алгоритмічна складність / Стек:** OpenCV (інтегрований через Kotlin Multiplatform или NDK), RANSAC (Random Sample Consensus) algorithm, Approximate Polygonal Contour Detection.
*   **Професійний академічний аргумент:** Ця функція є практичним демонстраційним прикладом застосування одного з фундаментальних алгоритмів комп'ютерного зору — RANSAC. Він широко використовується для оцінки параметрів математичної моделі з наявністю викидів (outliers), що є типовим для даних, отриманих від людини (рух руки). Реалізація цього алгоритму в освітньому проєкті демонструє розуміння методів відсіювання шуму та апроксимації даних, що є ключовими в Computer Vision.

*   **План інтеграції та алгоритм:**
    1.  **Інтеграція OpenCV:** Використовується бібліотека OpenCV, яку можна інтегрувати в проєкт через [Kotlin Multiplatform](https://github.com/korlibs/korge-samples/tree/master/native-opencv) для крос-платформенності або через Android NDK для максимального контролю та продуктивності.
    2.  **Збирання даних:** Штрихи, накреслені користувачем, представляються як послідовність точок `(x, y)` у відносних координатах полотна.
    3.  **Попередня обробка:** Вхідний `Mat` (двовимірний масив OpenCV) заповнюється штрихами. Застосовується алгоритм краєрозрізнення Canny для виділення контурів.
    4.  **Пошук контурів:** Використовується функція `Imgproc.findContours()` для отримання всіх замкнутих контурів на полотні.
    5.  **Апроксимація та розпізнавання:** Для кожного знайденого контуру застосовується алгоритм RANSAC для вибору найкращої геометричної моделі.
        *   **Для кіл:** `Imgproc.fitEllipse()` може дати еліпс, який далі перевіряється на відповідність колу.
        *   **Для багатокутників:** `Imgproc.approxPolyDP()` є ефективним для апроксимації контуру до багатокутника. Кількість вершин визначає тип фігури (3 - трикутник, 4 - прямокутник, >4 - інший багатокутник).
    6.  **Конвертація:** Результат розпізнавання (наприклад, центр, радіус, кут для кола) зберігається як новий векторний об'єкт (`CircleShape`, `RectangleShape`) та замінює оригінальні штрихи на полотні.
    7.  **Редактування:** Нові векторні об'єкти робляться трансформованими, що дозволяє користувачеві легко змінювати їх розмір та положення.

**Фіча 2: Function Plotter from Hand-drawn Graphs**

*   **Назва та суть:** Користувач намальовує графік функції (наприклад, параболу, синусоїду), і додаток спробує відтворити його у вигляді математичної функції `y = f(x)` та відобразити точний векторний графік.
*   **Алгоритмічна складність / Стек:** SymPy (Python-based math engine, інтегрований через Python-for-android або KMP), Least Squares Regression, Curve Fitting.
*   **Професійний академічний аргумент:** Ця функція демонструє інтеграцію з потужними зовнішніми математичними бібліотеками та застосування числових методів для розв'язання задачі апроксимації. Вона показує, як практична проблема (запис графіка) може бути зведена до математичної моделі, що є фундаментом для багатьох інженерних та наукових додатків.

*   **План інтеграції та алгоритм:**
    1.  **Інтеграція SymPy:** SymPy можна інтегрувати через бібліотеки, як-от `pytorch-android` або `buildozer` для створення готового APK. Це вимагає додаткової конфігурації, але дає доступ до повної потужності Python's scientific stack.
    2.  **Збирання та обробка точок:** Координати точок з накресленого користувачем штриха зчитуються. Точки впорядковуються по осі X, а аномальні значення (викиди) видаляються за допомогою простого фільтра.
    3.  **Апроксимація:** Використовується метод найменших квадратів (реалізований в SymPy) для підгонки рівняння до набору точок. Додаток може пробувати підігнати різні типи функцій:
        *   Поліноміальну регресію: `sympy.polys.polytools.polyfit`.
        *   Тригонометричну регресію для періодичних функцій.
    4.  **Відображення результату:** Результат підгонки (наприклад, `y = 2*x**2 + 3*x - 1`) виводиться на екран у форматі LaTeX. Далі, SymPy використовується для побудови точного векторного графіка цієї функції на Canvas, замінюючи накреслений штрих.

**Фіча 3: Handwriting to LaTeX Converter**

*   **Назва та суть:** Розпізнавання рукописного тексту та формул, введених на полотні, та їх конвертація в код LaTeX.
*   **Алгоритмічна складність / Стек:** TensorFlow Lite (для моделі розпізнавання символів), Custom-trained model (CROHME dataset), CNN (Convolutional Neural Network).
*   **Професійний академічний аргумент:** Ця функція є продемонстрованою практикою роботи з машинним навчанням на пристрої. Вона включає в себе вибір моделі (TFLite), підготовку набору даних (CROHME), тренування специфічної моделі для розпізнавання математичних символів та інтеграцію цієї моделі в Android-додаток. Це вражає викладачів, оскільки це реальний проєкт ML, а не просто виклик дощових бібліотек.

*   **План інтеграції та алгоритм:**
    1.  **Підготовка моделі:** Навчити легковагу нейронну мережу (CNN, наприклад, на базі MobileNet або ResNet) на наборі даних рукописних математичних символів, як-от CROHME. Модель навчається класифікувати кожен символ на 10x10 або 28x28 піксельну картинку.
    2.  **Інтеграція TFLite:** Модель `.tflite` додається до ресурсів додатку (`/src/main/assets`).
    3.  **Розділення символів:** Алгоритм розділяє накреслену строку на окремі символи. Це можна зробити, аналізуючи проміжки між контурами штрихів.
    4.  **Розпізнавання:** Кожен символ подається на вхід TFLite Interpreter, який повертає найімовірніший символ (наприклад, 'x', '+', '\alpha').
    5.  **Формування LaTeX:** Результати розпізнавання символів з'єднуються в логічну структуру. Для цього можна використовувати правила для обробки індексів (низьких та верхніх), дробів, коренів та інших математичних конструкцій, визначених за їхньою просторовою розташуванням на полотні.
    6.  **Експорт:** Сформований рядок LaTeX можна вивести на екран або експортувати для використання в документах (наприклад, у форматі Markdown або для подальшого відображення за допомогою бібліотеки MathJax).

Ці три функції разом створюють потужний освітній інструмент, який виходить далеко за межі простого малювального застосунку, пропонуючи студентам можливість експериментувати з алгоритмами Computer Vision, числовими методами та машинним навчанням.

## Каталог "Чіт-кодів" №2: Комп'ютерне зір та відтворення реальності

Цей розділ описує чотири інноваційні функції, що використовують передові технології комп'ютерного зору, відтворення реальності та теорії графів для створення унікальних освітніх можливостей. Кожна функція є комплексним проєктом, що вимагає глибокого технічного розуміння та інтеграції різних API та бібліотек.

**Фіча 4: Object Tracker with AR Overlays**

*   **Назва та суть:** Користувач намальовує силует об'єкта, і додаток відстежує руку або певну область у кадрі з камери та "накреслює" намальований об'єкт поверх реального світу, створюючи простий, але ефективний AR-ефект.
*   **Алгоритмічна складність / Стек:** MediaPipe Hands/Pose, CameraX, OpenGL ES / Custom Canvas, Augmented Reality (AR).
*   **Професійний академічний аргумент:** Ця функція є комплексним прикладом роботи з низькорівневими API Android для створення динамічного AR-додатку. Вона включає в себе використання системи відстеження об'єктів (MediaPipe) для отримання координат, роботу з камерою (CameraX) для отримання відеопотоку та відображення накладеного зображення (шляхом малювання на `GLSurfaceView` або власному `Canvas`). Це демонструє розуміння системних компонентів Android та графіки.

*   **План інтеграції та алгоритм:**
    1.  **Інтеграція бібліотек:** Додати залежності для CameraX (`camera-camera2`, `camera-lifecycle`, `camera-view`), MediaPipe (Hand Tracking або Pose), та OpenGL ES.
    2.  **Вибір об'єкта:** Користувач намальовує силует на основному Canvas. Цей силует зберігається як `Path` або конвертується в `Bitmap`.
    3.  **Запуск AR-режиму:** При вході в AR-режим додаток відкриває відображення камери за допомогою `PreviewView` з CameraX.
    4.  **Відстеження:** MediaPipe стартує потік обробки кадрів з камери. Він визначає ключові точки тіла (наприклад, середину долоні) та повертає їхні 2D координати в кадрі.
    5.  **Масштабування та позиціонування:** Координати, отримані від MediaPipe, використовуються для позиціонування намальованого об'єкта. Необхідно враховувати розмір екрану та розгортання камери для коректного масштабування.
    6.  **Відображення:** Намальований об'єкт малюється поверх відеокадру. Це можна зробити, використовуючи `GLSurfaceView` з власним шейдером, який накладає текстуру (намальований об'єкт) на поверхню, або створивши спеціальний `Compose` Composable, що відображається поверх `Activity` за допомогою `OverlayView`.

**Фіча 5: Graph Schema Analyzer**

*   **Назва та суть:** Користувач намальовує схему графа (точки як вузли, стрілки як ребра), і додаток автоматично аналізує її, перетворюючи на формальні структури даних, такі як матриця суміжності та JSON, та проводить базовий аналіз (наприклад, знаходження компонент зв'язності).
*   **Алгоритмічна складність / Стек:** Custom algorithms, Hough Circle Transform, Line Detection (OpenCV), Breadth-First Search (BFS), Depth-First Search (DFS).
*   **Професійний академічний аргумент:** Ця функція є чудовим прикладом міжгалузевого застосування алгоритмів. Вона об'єднує обробку зображень (знаходження геометричних фігур на малюнку) з алгоритмами теорії графів. Це ілюструє, як комп'ютерне зір може бути використаний для інтерпретації діаграм та їхнього перетворення в машинно-читану форму, що є важливою задачею в багатьох освітніх та інженерних сферах.

*   **План інтеграції та алгоритм:**
    1.  **Інтеграція OpenCV:** Як і в попередній фічі, OpenCV використовується для обробки зображення полотна.
    2.  **Виділення вузлів:** Зображення полотна перетворюється в чорно-білий формат. Алгоритм Hough Circle Transform (`Imgproc.HOUGH_GRADIENT`) використовується для пошуку окружностей, які представляють вузли графа.
    3.  **Виділення ребер:** Алгоритм Canny використовується для пошуку країв. Потім `Imgproc.HoughLinesP()` використовується для пошуку коротких ліній, які можуть представляти ребра. Для кожної знайденої лінії визначаються її кінцеві точки.
    4.  **Побудова графа:** На основі координат вузлів та кінців відрізків будується внутрішнє представлення графа (наприклад, список суміжності). Вузли вважаються однаковими, якщо їхні центри знаходяться на невеликій відстані один від одного.
    5.  **Аналіз:** Застосовуються класичні алгоритми обходу графа (BFS або DFS) для проведення аналізу:
        *   **Знаходження компонент зв'язності:** BFS/DFS використовується для підрахунку кількості окремих підграфів.
        *   **Пошук циклів:** Модифікований DFS може використовуватися для виявлення циклів.
    6.  **Експорт:** Додаток виводить результати аналізу та пропонує експорт графа у вигляді матриці суміжності та JSON-структури, яка описує вузли та ребра.

**Фіча 6: Smart Lecture Recorder**

*   **Назва та суть:** Синхронізація кожного штриха на полотні з відповідним моментом аудіозапису лекції. Користувач може натиснути на будь-який штрих у списку та перемотати аудіозапис до того моменту, коли цей штрих був намальований.
*   **Алгоритмічна складність / Стек:** Oboe (низьколатентний аудіо), Coroutines, Timestamping, Shared Preferences / Room Database.
*   **Професійний академічний аргумент:** Ця функція розв'язує складну проблему синхронізації різних потоків даних (звук, графіка) в реальному часі. Вона демонструє використання низьколатентних аудіо-бібліотек (Oboe) та архітектурні патерни для ефективного зберігання та пошуку даних. Це актуальна тема у розробці медіа-додатків та систем для освіти.

*   **План інтеграції та алгоритм:**
    1.  **Інтеграція Oboe:** Як і в бузі #3, використовується Oboe для запису аудіо з мінімальною затримкою.
    2.  **Структура даних:** Створюється клас `StrokeWithTimestamp`, що містить дані про штрих (`Stroke`) та точний час його створення (в мілісекундах).
    3.  **Запис:** Під час запису аудіо, для кожного нового штриха, доданого до полотна, його даним та поточний час з API `System.currentTimeMillis()` додаються до списку `strokeList`.
    4.  **Збереження:** Цей список разом з початковим часом запису зберігається у файл або базу даних (Room).
    5.  **Відтворення:** При відтворенні аудіо, додаток відстежує поточний час відтворення та знаходить відповідний штрих для відображення на екрані.
    6.  **Навігація:** Створюється інтерфейс (наприклад, вертикальний список), де кожен елемент відповідає штриху. Натискання на елемент викликає перемотку аудіозапису до збереженого часового мітки.

**Фіча 7: Physics-based Object Dynamics**

*   **Назва та суть:** Намальовані фігури (прямокутники, кола) конвертуються в фізичні об'єкти з масою, формою та матеріальними властивостями, які потрапляють в симуляцію 2D-гравітації та взаємодіють між собою та зі стінками полотна.
*   **Алгоритмічна складність / Стек:** Box2D (2D фізичний рушій), JNI/Native Libraries.
*   **Професійний академічний аргумент:** Ця функція є вражаючим прикладом інтеграції потужного зовнішнього нативного бібліотеки (Box2D) в Android-додаток. Вона демонструє, як можна перетворити статичні графічні примітиви в динамічні об'єкти, що підкоряються законам фізики. Це є основою для багатьох ігор та симуляторів, і його реалізація в освітньому проєкті показує високий рівень технічної майстерності.

*   **План інтеграції та алгоритм:**
    1.  **Інтеграція Box2D:** Додати Box2D до проєкту через NDK. Це вимагає створення JNI-обгортки для взаємодії Kotlin-коду з C++.
    2.  **Конвертація:** Коли користувач намальовує фігуру, додаток створює її фізичну аналогію. Наприклад, прямокутник на Canvas перетворюється в `PolygonShape` в Box2D. Розмір та положення визначають параметри тіла.
    3.  **Створення симуляції:** У фоновому потоці створюється екземпляр `World` з гравітацією. Цей `World` є центральним об'єктом, що керує всією симуляцією.
    4.  **Симуляція та оновлення:** У кожному кадрі (наприклад, в `onDraw` або через `Handler`) викликається метод `world.step()`, який розраховує новий стан всіх тіл.
    5.  **Відображення:** Для кожного тіла в `World` додаток читає його оновлене положення та орієнтацію та оновлює відповідний графічний об'єкт на Canvas. Це створює ілюзію реального руху.
    6.  **Інтерактивність:** Користувач може взаємодіяти з фізичними об'єктами, наприклад, клікати на них, щоб додати силу.

Ці функції перетворюють Sketchpad на багатофункціональний платформу для експериментів, що є надзвичайно цінною для студентів, які хочуть попрактикуватися в застосуванні передових інженерних концепцій у практичному, візуальному контексті.

## Каталог "Чіт-кодів" №3: Аудіо-синхронізація, GPU-шейдинг та високопродуктивність

Цей розділ присвячений ще трьом високотехнологічним функціям, які роблять додаток Sketchpad не лише освітнім інструментом, але й демонстрацією передових можливостей графіки, звукової обробки та системного програмування на Android. Кожна функція є самодостатнім проєктом, що вимагає глибокого технічного поглиблення.

**Фіча 8: Oil Paint Shader Effect**

*   **Назва та суть:** Застосування реалістичного ефекту олійної живопису до штрихів, що створює ілюзію пінгвінів фарби та змішування кольорів.
*   **Алгоритмічна складність / Стек:** AGSL (Android Graphics Shader Language), RenderEffect, RuntimeShader, Perlin Noise.
*   **Професійний академічний аргумент:** Ця функція демонструє використання GPU-шейдерів для створення складних візуальних ефектів, що є високопродуктивним, оскільки обчислення виконуються паралельно на графічній картці. Вона включає в себе написання власного шейдера на AGSL, який є стандартом Android для написання шейдерів. Це показує знання сучасних технік GPU-programming у контексті Jetpack Compose.

*   **План інтеграції та алгоритм:**
    1.  **Створення AGSL шейдера:** Створюється файл `oil_paint_shaderagal` у каталозі `res/raw/`. Цей файл містить код шейдера на AGSL.
    2.  **Логіка шейдера:** Шейдер отримує на вхід вхідне зображення (текстуру) та координати поточного пікселя.
        ```agsl
        uniform float2 iResolution;
        uniform half2 iMouse;
        uniform float iTime;
        uniform sampler sTexture;

        float hash(float2 p) {
            // Проста функція хешування для генерації шуму
            return fract(sin(dot(p, float2(12.9898, 78.233))) * 43758.5453);
        }

        half4 main(sampler sTexture) {
            float2 uv = getLocalCoord();
            half4 color = fetch(sTexture, uv);

            // Додаємо "пінгвіни" фарби шляхом випадкового зміщення
            float noise = hash(uv * 100.0 + iTime);
            float2 offset = (float2(noise, hash(uv * 0.1)) - 0.5) * 0.02;
            color = fetch(sTexture, uv + offset);

            // Додаємо розмиття для ефекту змішування кольорів
            half4 sum = half4(0.0);
            int count = 0;
            for(int i = -1; i <= 1; ++i) {
                for(int j = -1; j <= 1; ++j) {
                    sum += fetch(sTexture, uv + float2(i, j) * 0.01);
                    count++;
                }
            }
            color = sum / count;

            return color;
        }
        ```
    3.  **Інтеграція в Compose:** У Kotlin-коді завантажується шейдер та застосовується до `Brush` за допомогою `RenderEffect`.
        ```kotlin
        val shader = RuntimeShader.load(context.resources, R.raw.oil_paint_shader)
        val renderEffect = RenderEffect(shader)

        // Використання
        drawCircle(
            color = Color.White,
            radius = 100f,
            brush = Brush.radialGradient(
                colors = listOf(Color.Red, Color.Yellow),
                center = Offset(100f, 100f),
                radius = 100f,
                renderEffect = renderEffect
            )
        )
        ```
Цей підхід дозволяє створювати динамічні та високопродуктивні візуальні ефекти, які було б дуже важко реалізувати на стороні CPU [[87,189]].

**Фіча 9: Watercolor Bleed Simulation**

*   **Назва та суть:** Реалістична модель розмазування фарби по паперу, де кольори з різних штрихів змішуються на межах контакту.
*   **Алгоритмічна складність / Стек:** AGSL, Partial Differential Equations (PDE), Framebuffers (rendertargets).
*   **Професійний академічний аргумент:** Це одна з найбільш вражаючих функцій, що може бути реалізована. Вона демонструє застосування PDE, що є фундаментальним інструментом у фізиці та інженерії, для створення реалістичного ефекту в побутовому додатку. Реалізація симуляції на GPU за допомогою AGSL є високопродуктивним рішенням.

*   **План інтеграції та алгоритм:**
    1.  **Модель:** Використовується проста модель розповсюдження, схожа на рівняння дифузії: $ \frac{\partial C}{\partial t} = D \nabla^2 C $, де C - концентрація фарби, D - коефіцієнт дифузії.
    2.  **Двійкові буфери:** Для симуляції використовуються два буфери (рендер-таргети): один для поточного стану (`currentColorBuffer`), другий для наступного (`nextColorBuffer`).
    3.  **AGSL Шейдер (PDE Solver):** Шейдер на AGSL виконує один крок симуляції. Він проходить по `currentColorBuffer`, розраховує новий стан для кожного пікселя на основі середнього значення кольору в сусідніх пікселях та пише результат у `nextColorBuffer`.
        ```agsl
        uniform sampler sCurrent;
        uniform sampler sPrevious; // Для додавання нової фарби

        half4 main(sampler sCurrent, sampler sPrevious) {
            float2 uv = getLocalCoord();
            half4 currentColor = fetch(sCurrent, uv);
            half4 sumNeighbors = half4(0.0);

            // Проста апроксимація Laplacian: середнє значення сусідів
            sumNeighbors += fetch(sCurrent, uv + float2(1.0, 0.0));
            sumNeighbors += fetch(sCurrent, uv + float2(-1.0, 0.0));
            sumNeighbors += fetch(sCurrent, uv + float2(0.0, 1.0));
            sumNeighbors += fetch(sCurrent, uv + float2(0.0, -1.0));
            sumNeighbors /= 4.0;

            // Розрахунок нового кольору
            half4 newColor = lerp(currentColor, sumNeighbors, 0.1); // D*dt = 0.1

            // Додавання нової фарби з палітри
            half4 newInk = fetch(sPrevious, uv);
            if (newInk.a > 0.1) {
                newColor = newInk;
            }

            return newColor;
        }
        ```
    4.  **Ітерації:** Процес виконується декілька разів (наприклад, 5-10) для досягнення бажаного ефекту розмазування. Буфери міняються місцями на кожній ітерації.
Це створює динамічну та реалістичну інтерактивну поверхню, що вражає своєю технічною складністю.

**Фіча 10: Infinite Canvas with Virtual Memory Paging**

*   **Назва та суть:** Реалізація нескінченного холста, який не завантажує весь контент в оперативну пам'ять одночасно, а завантажує лише ту частину, яка потрапила в зону видимості камери.
*   **Алгоритмічна складність / Стек:** Custom data structure, BitmapPool, Lazy loading, Coordinate system transformation.
*   **Професійний академічний аргумент:** Ця функція є глибоким прикладом системного програмування та інженерії програмного забезпечення. Вона розв'язує важливу інженерну проблему масштабування за допомогою принципів віртуальної пам'яті, що є фундаментальним у системному програмуванні. Це показує здатність розробника створювати високопродуктивні системи, що можуть обходитися з великими обсягами даних.

*   **План інтеграції та алгоритм:**
    1.  **Розбиття на секції:** Холст розбивається на великі "секції" або "кластери" розміром, наприклад, 2048x2048 пікселів.
    2.  **Ліниве завантаження:** Тільки ті секції, які потрапили в зону видимості камери (визначеною поточною матрицею трансформації), завантажуються в `Bitmap`. Інші секції не завантажуються.
    3.  **Кешування:** Використовується пул `Bitmap` (`BitmapPool`) для повторного використання пам'яті під `Bitmap`. Коли секція виходить за межі видимості,
